## #7 — Local Speech-to-Text (Whisper/On-Device)

### The Problem

PointDev uses the Web Speech API (`webkitSpeechRecognition`), which sends audio to Google's servers for transcription. This is a **privacy liability** and a potential blocker for NLnet funding (NGI Zero prioritizes data sovereignty).

### Available Approaches

#### Option A: whisper.cpp via WebAssembly (Recommended)

**[ggml-org/whisper.cpp](https://github.com/ggml-org/whisper.cpp)** — C/C++ port of OpenAI's Whisper, runs in-browser via WASM.

- **Proven in browser:** Official WASM examples exist and work ([demo](https://ggml.ai/whisper.cpp/))
- **Model sizes:** tiny (75MB), base (142MB), small (466MB) — tiny is sufficient for real-time narration
- **Languages:** 50+ supported
- **Performance:** Near-native via WASM + WebGL/WebGPU acceleration
- **Privacy:** 100% on-device, zero network traffic
- **Latency:** ~1-3s for real-time, depending on model size and hardware

**Implementation path for PointDev:**
1. Compile whisper.cpp to WASM (or use pre-built binaries from the repo)
2. Load the tiny/base model on first use (~75-142MB download, cached)
3. Run in a Web Worker to avoid blocking the UI
4. Feed audio from `getUserMedia` → AudioWorklet → Whisper WASM worker
5. Return timestamped segments matching the current `VoiceSegment` interface

**Challenges:**
- Model download size (75MB minimum) — need a download progress UI
- First-run latency while model loads
- Audio resampling (Whisper expects 16kHz mono)
- No built-in interim results like Web Speech API — need to implement chunked processing

**Reference implementations:**
- [whisper.cpp WASM stream demo](https://ggml.ai/whisper.cpp/) — real-time browser transcription
- [SvelteKit + whisper.cpp integration](https://stackoverflow.com/questions/79406943) — worker thread setup
- [Handy](https://www.reddit.com/r/LocalLLaMA/comments/1ldvosh/) — open source offline STT app using whisper.cpp

#### Option B: Chrome's Built-in Prompt API (Gemini Nano)

Chrome 139+ ships Gemini Nano on-device. The Prompt API is available to extensions.

- **Zero model download** — already on the device
- **Hardware accelerated** — uses device GPU
- **Free, no API key**
- **Limitations:** Text-only for now (no audio input). The Prompt API accepts text/image/audio inputs as of Chrome 139 origin trial, but audio support is experimental and not yet stable for extensions.

**Verdict:** Not viable for speech-to-text yet. Watch for future Chrome releases.

#### Option C: Transformers.js (Hugging Face)

[Transformers.js](https://github.com/xenova/transformers.js) runs Whisper models in the browser via ONNX Runtime (WASM/WebGPU).

- **Easier integration** than raw whisper.cpp WASM
- **npm package** (`@xenova/transformers`) — familiar JS API
- **Whisper models available** on Hugging Face Hub
- **Performance:** Slightly slower than whisper.cpp WASM due to ONNX overhead
- **Size:** Still requires model download

**Verdict:** Good alternative if whisper.cpp WASM integration proves complex.

### Recommended Strategy

1. **Keep Web Speech API as default** — it works, it's instant, no download
2. **Add whisper.cpp WASM as opt-in "Privacy Mode"** — user toggles in settings
3. **Show clear UI** explaining the tradeoff: "Fast (Google)" vs "Private (On-device)"
4. **Progressive enhancement:** Download model in background after first use
5. **Future:** When Chrome's Prompt API supports audio input natively, switch to that

### Architecture Impact

```
Current:  getUserMedia → SpeechRecognition → onresult → VoiceSegment
New:      getUserMedia → AudioWorklet → Worker(whisper.wasm) → VoiceSegment
```

Both paths produce the same `VoiceSegment` interface — no changes needed downstream.

## #10 — Pluggable Output Formats (JSON, Markdown, MCP)

### The Problem

PointDev compiles capture sessions into a single plain-text template format. To integrate with the AI tools ecosystem, it needs structured output formats — especially MCP (Model Context Protocol), which is becoming the standard for AI tool interop.

### The MCP Ecosystem (2026 State)

MCP is now the dominant protocol for connecting AI tools to external data. Key players:

| Tool | MCP Role | Stars |
|------|----------|-------|
| **Chrome DevTools MCP** (Google) | Official Chrome debugger for AI agents | — |
| **BrowserTools MCP** (AgentDesk) | Console/network/screenshot capture | 7.1k |
| **mcp-pointer** (etsd-tech) | DOM element selection for coding tools | — |
| **mcp-chrome** (hangwin) | Full browser automation via MCP | 10.9k |
| **Browser MCP** (Chrome Web Store) | Browser automation for VS Code/Cursor | 100k users |

**PointDev's opportunity:** None of these capture *human intent* (voice, annotations, dwell). PointDev as an MCP server would be the only one providing structured developer feedback, not just browser state.

### Output Format Strategy

#### Format 1: JSON (structured data)

```json
{
  "context": { "url": "...", "title": "...", "viewport": {...} },
  "voice": { "segments": [{ "text": "...", "startMs": 0, "endMs": 1000 }] },
  "annotations": [{ "type": "circle", "target": ".btn", "timestamp": 5000 }],
  "screenshots": [{ "trigger": "voice", "score": 0.7, "dataUrl": "..." }],
  "cursor": { "dwells": [{ "element": ".nav", "durationMs": 3200 }] }
}
```

Use case: Programmatic consumption, piping into other tools, custom integrations.

#### Format 2: Markdown (human-readable)

The current template format is essentially Markdown already. Formalize it with proper headers, tables, and image embeds.

Use case: GitHub issues, Linear tickets, documentation, direct paste into any tool.

#### Format 3: MCP Server (live integration)

PointDev runs a local MCP server that exposes capture data as MCP resources and tools.

**Architecture:**

```
PointDev Extension ←→ Local MCP Server (localhost:3456)
                          ↕
                    AI Coding Tool (Claude Code, Cursor, etc.)
```

**MCP Resources (read-only data):**
- `pointdev://session/current` — current capture session
- `pointdev://session/{id}` — historical session
- `pointdev://screenshots/{id}` — screenshot images

**MCP Tools (actions):**
- `start_capture` — begin a new session
- `stop_capture` — end and compile
- `get_session` — retrieve structured session data

**Implementation options:**
1. **Node.js MCP server** — separate process, communicates with extension via Native Messaging or WebSocket
2. **Built into the extension** — extension hosts the MCP server directly (limited by Chrome's networking APIs)
3. **Hybrid** — extension captures, writes to a shared file/port, MCP server reads from it

Option 1 (Node.js) aligns with how BrowserTools MCP works (separate `npx` server).

### Reference Implementations

- **BrowserTools MCP architecture:** Chrome extension → WebSocket → Node.js server → MCP server. Three components, but the pattern is proven and adopted by 7k+ users.
- **mcp-pointer:** Chrome extension + local MCP server for DOM element pointing. Simpler architecture, good reference for PointDev's scope.
- **chrome-devtools-mcp (Google):** Official, uses Chrome DevTools Protocol. More complex but authoritative.

### Recommended Strategy

1. **Phase 1 (now):** Add JSON export alongside the current text format. Button in OutputView: "Copy as JSON"
2. **Phase 2:** Add Markdown export with embedded screenshot references
3. **Phase 3:** Build MCP server (Node.js process) that reads capture data from the extension via Native Messaging

## #12 — Direct Delivery to AI Tools (Bridge Server)

### The Problem

Users currently copy PointDev output to clipboard and paste into Claude Code / Cursor. This friction breaks flow and loses screenshot images (clipboard can't carry structured data + images together).

### Existing Bridge Patterns

#### Pattern A: BrowserTools MCP (WebSocket + Node.js)

**[AgentDeskAI/browser-tools-mcp](https://github.com/AgentDeskAI/browser-tools-mcp)**

Architecture:
```
Chrome Extension ←WebSocket→ Node.js Server (port 3025) ←stdio→ MCP Server
                                                                    ↕
                                                              Claude/Cursor
```

- Extension captures console logs, network, screenshots, selected elements
- Node.js server aggregates and truncates data to avoid token limits
- MCP server exposes tools: `getConsoleLogs`, `getNetworkLogs`, `takeScreenshot`, `getSelectedElement`
- **Install:** `npx @agentdeskai/browser-tools-server` + `npx @agentdeskai/browser-tools-mcp`

**Pros:** Proven (7k stars), well-documented, works with Cursor/Claude Desktop/Cline.
**Cons:** Requires running two `npx` commands, three components total.

#### Pattern B: mcp-pointer (Native Messaging)

**[etsd-tech/mcp-pointer](https://github.com/etsd-tech/mcp-pointer)**

Architecture:
```
Chrome Extension ←Native Messaging→ Local MCP Server
                                        ↕
                                  Claude Code / Cursor
```

- User Option+Clicks a DOM element in the browser
- Extension sends element data (CSS, attributes, position) via Native Messaging
- MCP server exposes it as a tool: AI can query what element was pointed at

**Pros:** Simpler than BrowserTools (two components, not three). Native Messaging is more reliable than WebSocket.
**Cons:** One-way (element data only), no screenshots or voice.

#### Pattern C: chrome-extension-bridge-mcp (WebSocket)

Simple WebSocket bridge between a Chrome extension and MCP clients. Lightweight, generic.

#### Pattern D: File-based Bridge

Extension writes capture data to a JSON file in a known location. MCP server watches the file.

```
Extension → writes to ~/.pointdev/sessions/current.json
MCP Server → watches file, serves via MCP
```

**Pros:** Simplest possible architecture. No WebSocket, no Native Messaging. Works even if the server restarts.
**Cons:** File I/O latency (~100ms), can't stream real-time data, screenshots as separate files.

### Recommended Strategy for PointDev

**Hybrid: Native Messaging for data, file system for screenshots**

```
PointDev Extension
    ↓ Native Messaging (JSON)
PointDev MCP Server (Node.js, installed via npx)
    ↓ stdio
Claude Code / Cursor / Any MCP Client
```

1. **Extension** captures session data and sends via Chrome Native Messaging to the local server
2. **Server** receives structured session data, stores screenshots to temp directory
3. **MCP server** exposes tools:
   - `get_current_session` — returns the full capture session as JSON
   - `get_screenshot(index)` — returns screenshot image
   - `get_voice_transcript` — returns timestamped segments
   - `get_annotations` — returns annotation details with element context
4. **AI tool** queries these tools directly — no copy-paste needed

**Why Native Messaging over WebSocket:**
- Chrome-native, no port conflicts
- Extension ↔ native app communication is a first-class Chrome API
- More reliable than WebSocket (no reconnection handling)
- Security: only the registered native app can receive messages

**Install experience:**
```bash
npx @pointdev/bridge  # Installs native messaging host + MCP server
```

### Zero-opCost Consideration

All three approaches (WebSocket, Native Messaging, file-based) are local-only with zero cloud cost. The MCP server is a local Node.js process. No API keys, no subscriptions.

## Summary: Implementation Priority

| Issue | Approach | Effort | Impact | Dependencies |
|-------|----------|--------|--------|-------------|
| **#7** Local STT | whisper.cpp WASM as opt-in privacy mode | Medium-High | High (NLnet) | None |
| **#10** Output formats | JSON export first, then MCP server | Low→Medium | High (ecosystem) | None for JSON; #12 for MCP |
| **#12** Bridge server | Native Messaging + MCP server | Medium | High (UX) | #10 (needs structured output) |

**Recommended order:**
1. **#10 JSON export** — smallest effort, unblocks #12
2. **#12 Bridge server** — biggest UX improvement, makes PointDev a live tool
3. **#7 Local STT** — biggest privacy win, addresses NLnet concern
